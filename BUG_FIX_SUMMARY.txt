 ROOT CAUSE: RACE CONDITION IN IPC HANDLERS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: windows/src/main/ipc/ipcHandlers.ts
Lines: 214-240

Problem:
  TWO handlers were listening to the SAME protocol messages:

  1. FileTransferManager (correct)
     └─ Processes messages properly
     └─ Calculates progress correctly
     └─ Emits 'transferUpdate' event

  2. ipcHandlers (REDUNDANT)
     └─ ALSO listens to raw protocol messages
     └─ Hardcodes status values
     └─ Sends conflicting updates to UI

Result:
  Race condition! Both handlers fire at same time with different data.
  Android gets conflicting updates → Progress stuck at 0%


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 THE FIX
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Removed 27 lines of redundant code from ipcHandlers.ts

BEFORE:
┌────────────────────────────────────────────────────────────────┐
│ connectionManager.on(`message:FILE_TRANSFER_REQUEST`)          │
│   └─ win.webContents.send('file:transferUpdate', {            │
│        status: 'pending'  ❌ HARDCODED                         │
│      })                                                         │
│ connectionManager.on(`message:FILE_TRANSFER_PROGRESS`)         │
│   └─ win.webContents.send('file:transferUpdate', {            │
│        status: 'transferring'  ❌ HARDCODED                    │
│      })                                                         │
│ connectionManager.on(`message:FILE_TRANSFER_COMPLETE`)         │
│   └─ win.webContents.send('file:transferUpdate', {            │
│        status: 'completed'  ❌ HARDCODED                       │
│      })                                                         │
└────────────────────────────────────────────────────────────────┘

AFTER:
┌────────────────────────────────────────────────────────────────┐
│ fileTransferManager.on('transferUpdate', (transfer) => {       │
│   BrowserWindow.getAllWindows().forEach(win => {              │
│     win.webContents.send('file:transferUpdate', transfer);   │
│     ✅ Uses actual calculated state                            │
│   });                                                           │
│ });                                                             │
└────────────────────────────────────────────────────────────────┘

KEY CHANGE:
  ❌ Raw protocol messages → Multiple handlers → Conflicts
  ✅ Domain model (FileTransferManager) → Single handler → Consistency
